# 研究计划：ConSC-Verify

## 研究目标

设计并评估 **ConSC-Verify（Consistency-Calibrated Selective Verification）**：利用黑盒大语言模型可获得的风险信号，对高风险回答选择性启动验证，而不是像全量 CoVe 一样对每题执行验证。目标是在事实性接近 CoVe-all 的条件下，降低平均 API 调用、token 和延迟。

这是一项“方法设计 + 严格实证验证”工作；不宣称首次提出自验证或检索纠错。

## 原有工作的继承

- 阶段一：模型原理、幻觉分类与 CoVe 文献调研，成为相关工作。
- 阶段二：API 调用、GAOKAO-Bench 解析与逐题评测，成为实验工程基础。
- 阶段三：低置信度触发二次确认的想法，成为 ConSC-Verify 的核心。

报告只保留作者本人能够通过代码提交、实验日志或笔记佐证的内容。

## 数据集职责

| 角色 | 数据集 | 结论边界 |
|---|---|---|
| 主实验 | FactBench | 真实交互风格下的事实性；报告 Supported / Unsupported / Undecidable。 |
| 压力测试 | KG-FPQ | 面对错误前提时是否纠错、拒答或顺着编造。 |
| 开发与中文辅助 | Chinese SimpleQA | 选择风险阈值，并检查中文短事实问答的可迁移性。 |
| 早期探索 | GAOKAO-Bench、TruthfulQA | 只用于说明问题如何被发现，不得与最终结果混写。 |

## 方法定义

1. **Direct**：一次回答。
2. **Challenge**：一次回答后，要求审视事实和推理并重答。
3. **CoVe-all**：每题执行草稿、验证问题、独立验证答案、重写四步。
4. **ConSC-Verify（本文方法）**：
   - 先得到初答；
   - 计算风险分数；
   - 低风险直接输出；高风险执行 CoVe 验证；
   - 验证后仍冲突或缺少依据时，输出明确的不确定性说明，而不是编造。

风险信号按接口能力二选一：

- 若 API 支持 `logprobs`：使用答案 token 的不确定性或选项间概率间隔；
- 否则：以同题多次采样的一致性作为黑盒风险代理，并额外标记格式违规和空答案。

开发集用于选择风险阈值；测试集不得参与阈值、提示词、样本过滤或评分规则的选择。

## 本轮固定模型

主实验使用官方 API 的 `deepseek-v4-flash`，`base_url` 为 `https://api.deepseek.com`。基线、Challenge、CoVe-all 和 ConSC-Verify 均使用相同模型、同一非思考模式和相同输出上限；不再使用即将停用的 `deepseek-chat` 或 `deepseek-reasoner` 别名。

选择 Flash 的原因是本研究包含多次验证调用，必须同时报告方法效果和成本。若主实验完成后仍有预算，再在冻结的测试子集上使用 `deepseek-v4-pro` 复验方法趋势；该复验不替代主结果。首次 20 题冒烟测试需确认 API 是否实际返回 `logprobs`。若未返回或调用报错，切换到多次采样一致性分支，并在日志中记录这一接口限制。

冒烟测试已确认 `logprobs` 返回，因此本轮选择概率校准分支：从首个答案 token 的选中 log 概率、与最强替代 token 的概率间隔、以及格式违规中抽取风险特征。风险阈值由开发集选择为“在达到预设错误召回率的阈值中最高者”，以优先减少不必要的验证调用；测试集不参与该选择。

## 评价指标

- 事实性：准确率，或 Supported / Unsupported / Undecidable 比例；
- 安全性：错误前提纠正率、错误前提顺从率、拒答率；
- 风险识别：触发率、错误回答召回率、错误回答漏检率；
- 效率：平均 API 调用次数、输入/输出 token、平均延迟；
- 覆盖：有效作答率。拒答不能被当作正确答案，必须与覆盖率同时报告。

## 固定执行顺序

1. 检查数据许可、格式和数据版本；保存数据哈希。
2. 以 20 题完成端到端冒烟测试，检查逐题日志。
3. 在开发集冻结提示词、风险公式和阈值。
4. 依次运行 Direct、Challenge、CoVe-all、ConSC-Verify。
5. 对同一批测试题做配对比较；记录成本与延迟。
6. 人工复核边界样本和方法改变结论的样本，形成错误类型案例。
7. 公开代码、配置模板、数据说明、去敏后的逐题日志与绘图脚本。
